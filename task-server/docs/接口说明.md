# 任务管理接口说明

## 基础说明

当前后端使用 NestJS、Prisma 和 PostgreSQL 实现任务管理接口。

用户需要先注册或登录，拿到 JWT `accessToken` 后，才能访问任务接口。

任务数据按用户隔离：

- `GET /task` 只返回当前登录用户的任务。
- 查询、更新、删除、状态流转任务时，只能操作当前登录用户自己的任务。
- 如果访问别人的任务，接口会按“任务不存在”处理，返回 `404`。

## 基础地址

```text
http://localhost:3000
```

## Postman 使用流程

1. 调用 `POST /auth/register` 注册用户。
2. 调用 `POST /auth/login` 登录。
3. 从登录响应中复制 `data.accessToken`。
4. 请求任务接口时，在 `Headers` 中新增：

```text
Authorization: Bearer accessToken
```

示例：

```text
Authorization: Bearer eyJhbGciOiJIUzI1NiIsInR5cCI6...
```

如果任务接口没有携带 token，或 token 错误、过期，会返回 `401`。

## 数据结构

### User

```ts
interface User {
  id: string
  email: string
  nickname: string | null
  createdAt: string
  updatedAt: string
}
```

说明：

- 密码不会在接口响应中返回。
- 数据库中保存的是加密后的密码 hash，不保存明文密码。

### TaskItem

```ts
interface TaskItem {
  id: string
  title: string
  description: string
  status: 'todo' | 'doing' | 'done'
  level: 'high' | 'medium' | 'low'
  createdAt: string
  updatedAt: string
  userId: string
}
```

说明：

- `status`：任务状态。
  - `todo`：待处理
  - `doing`：进行中
  - `done`：已完成
- `level`：任务优先级。
  - `high`：高
  - `medium`：中
  - `low`：低
- `createdAt` 和 `updatedAt` 在接口响应中使用 ISO 时间字符串。
- `userId` 表示任务所属用户，由后端根据当前登录用户自动绑定。

## 通用认证说明

除注册和登录接口外，任务接口都需要请求头：

| 请求头 | 必填 | 说明 |
| --- | --- | --- |
| Authorization | 是 | JWT 认证信息，格式为 `Bearer accessToken` |

未登录或 token 无效时返回：

```json
{
  "message": "Unauthorized",
  "statusCode": 401
}
```

## 1. 注册

### 请求方法

```text
POST
```

### 请求路径

```text
/auth/register
```

### 请求参数

无。

### 请求体

| 字段名 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| email | string | 是 | 邮箱 |
| password | string | 是 | 密码，至少 6 个字符 |
| nickname | string | 否 | 昵称 |

示例：

```json
{
  "email": "user@example.com",
  "password": "123456",
  "nickname": "小豪"
}
```

### 响应数据

```json
{
  "code": 200,
  "message": "注册成功",
  "data": {
    "id": "user-id",
    "email": "user@example.com",
    "nickname": "小豪",
    "createdAt": "2026-06-29T00:00:00.000Z",
    "updatedAt": "2026-06-29T00:00:00.000Z"
  }
}
```

### 错误响应

邮箱已被注册时返回：

```json
{
  "message": "邮箱已被注册",
  "error": "Conflict",
  "statusCode": 409
}
```

```json
{
  "code": 200,
  "message": "登录成功",
  "data": {
```

- `accessToken` 是后续请求任务接口要使用的 token。

邮箱不存在或密码错误时返回：

```json
{
  "message": "邮箱或密码错误",
  "error": "Unauthorized",
```

## 9. 流转到下一个状态

### 请求方法

```text
PATCH
```

### 请求路径

```text
/task/:id/status/next
```

### 请求头

| 请求头 | 必填 | 示例 |
| --- | --- | --- |
| Authorization | 是 | `Bearer accessToken` |

### 请求参数

| 参数名 | 位置 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- | --- |
| id | path | string | 是 | 任务 ID |

### 请求体

无。

### 响应数据

```json
{
  "code": 200,
  "message": "状态更新成功",
  "data": {
    "id": "task-id",
    "title": "完成任务管理页面",
    "description": "实现任务列表、筛选和状态切换",
    "status": "doing",
    "level": "high",
    "createdAt": "2026-06-26T08:00:00.000Z",
    "updatedAt": "2026-06-26T09:10:00.000Z",
    "userId": "user-id"
  }
}
```

说明：

- 根据任务当前状态，自动流转到下一个状态。
- 状态流转规则：
  - `todo` -> `doing`
  - `doing` -> `done`
  - `done` -> `todo`
- 修改后数据库会自动刷新 `updatedAt`。
- 只能修改当前登录用户自己的任务。

### 错误响应

任务不存在，或任务不属于当前用户时返回：

```json
{
  "message": "任务不存在",
  "error": "Not Found",
  "statusCode": 404
}
```

## 参数校验规则

当前 DTO 和 `ValidationPipe` 按下面规则校验：

| 字段名 | 校验规则 |
| --- | --- |
| email | 必填，必须是邮箱格式 |
| password | 必填，字符串；注册时至少 6 个字符 |
| nickname | 可选，字符串 |
| title | 必填，字符串，不能为空 |
| description | 必填，字符串，不能为空 |
| level | 必须是 `high`、`medium`、`low` 之一 |
| status | 必须是 `todo`、`doing`、`done` 之一 |

参数错误时返回：

```json
{
  "message": [
    "任务标题不能为空"
  ],
  "error": "Bad Request",
  "statusCode": 400
}
```
